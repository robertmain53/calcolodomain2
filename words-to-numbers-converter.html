Data Source & Methodology

This tool implements short-scale English number naming (million=106, billion=109, trillion=1012, …) and accepts optional British-style “and” constructions. For editorial conventions around numerals and hyphenation, see:

  • U.S. Government Publishing Office Style Manual (2016)
  • Long and short scales (overview)

The “Formula” Explained

Tokenization: Split words into tokens \( t_i \) and map to values.
Aggregation: For a segment \( S \), compute
segment = sum(small_numbers) + (hundreds × 100)
When a scale word \( M \in \{10^3,10^6,10^9,\dots\} \) appears, apply:
accumulator = accumulator + segment × M;  segment = 0
Final value:
value = sign × (accumulator + segment + decimal_part)

We parse from left to right, folding “hundred” within each segment and multiplying at “thousand/million/…”.

Glossary of Inputs & Outputs

  • Number in words: English phrase describing a number (e.g., “two million and ten”).
  • Allow “and”: Accepts “and” within hundreds (e.g., “one hundred and five”).
  • Short scale: Uses million/billion/trillion… (US/modern international).
  • Normalize ordinals: Maps simple ordinals (e.g., “twenty-first”) to cardinals.
  • Numeric value: Parsed decimal number (up to 1018 in magnitude).
  • Normalized words: Cleaned, canonicalized version of the input phrase.
  • Sign: positive or negative.
  • Magnitude: Approximate order (e.g., thousands, millions).

How It Works: A Step-by-Step Example

Input: negative one hundred twenty-three thousand and forty-five point six seven

  1. Sign = negative → -1.
  2. Segments: one hundred twenty-three → 123; “thousand” → multiply segment by 1,000 → 123,000.
  3. Remainder: forty-five → 45; add to total → 123,045.
  4. Decimal: point six seven → 0.67.
  5. Result: -123045.67

Frequently Asked Questions

Which English variant is this based on?

Short-scale English (US/modern international). Optional “and” is accepted for British-style phrasing.

What’s the largest number supported?

Up to one quintillion (1018) in absolute value. Larger inputs are rejected to prevent overflow.

Does it handle decimals and negative values?

Yes. Use “point” for decimals and “minus” or “negative” for the sign.

Can I paste noisy text (e.g., extra punctuation)?

Non-number words and extraneous punctuation are ignored where safe. Ambiguities will trigger validation errors.

Are ordinals like “twenty-first” supported?

Basic ordinal normalization maps common ordinals to cardinals before parsing.

How do you validate correctness?

The parser follows deterministic folding rules for hundreds and scale words, unit-tested against edge cases (hyphens, “and”, decimals, negatives).
