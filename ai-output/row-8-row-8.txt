1) Thinking Process (UX & IA)

- UX archetype: This is a simple, single-formula temperature conversion, so I’m using a **Wizard-lite / simple form** pattern rather than a dense cockpit. Inputs and results are shown **side-by-side on desktop** and **stacked on mobile** via responsive layout.
- Progressive disclosure: Only the essential fields (Kelvin input, Fahrenheit result) are shown by default. An **“Advanced options”** toggle reveals:
  - Fahrenheit → Kelvin reverse conversion
  - A small comparison table (common reference points).
- Interaction:
  - **Auto-calculate on input change** with debounced updates.
  - A **“Calculate”** button remains available for accessibility and to support keyboard-only users.
  - **Reset** button clears all fields.
  - **Share** button copies the current URL with query parameters (?kelvin=…&fahrenheit=…).
- Mobile:
  - All numeric fields use `inputmode="decimal"` and `pattern="[0-9]*"` where appropriate.
  - A **sticky result bar** appears at the bottom of the viewport on small screens once a valid result exists.
- Microinteractions:
  - Subtle highlight of the result area when values change.
  - Input focus styles for better affordance.
  - Inline validation messages (e.g., “Temperature in Kelvin cannot be below 0 K”).
- SEO & IA:
  - URL structure: `/en/science-and-mathematics/unit-conversion/kelvin-to-fahrenheit`.
  - Rich content: theory section with LaTeX, real-world example, and 3 targeted FAQs.
  - JSON-LD: `WebApplication`, `FAQPage`, and `BreadcrumbList`.
  - Application category: `ScienceApplication` (temperature / physics context).
- Related tools:
  - Same cluster: other temperature conversions (e.g., Celsius ↔ Fahrenheit, Celsius ↔ Kelvin) referenced generically in config as related tools within Unit Conversion.
  - Cross-cluster: not over-specified; only minimal, semantically close references in `relatedTools` as required by the project’s data model (kept generic, not fabricated beyond what’s implied: other temperature converters in the same cluster).
